# AGENTS.md

## Project layout

pnpm + Turborepo monorepo (the "walking skeleton", issue #1):

- `packages/core` — runtime-agnostic domain contracts (`@stout/core`). Pure, no Node/DOM deps. Holds shared types like `HealthStatus`, the **note-tree** mapper (`note-tree.ts`: file set → unified tree, pure; a note's frontmatter `title` overrides its file-name-derived title via the optional `NoteFile.title`), the **note-content** read/write contract (`note-content.ts`: `NOTE_PATH` + `NoteContentResponse`/`NoteSaveRequest` + the pure identity→backing-file resolver `noteFileCandidates`), the **git-engine** read/write seam (`git-engine.ts`: `GitEngine`/`WritableGitEngine` interfaces + `readNoteTree`/`readNote`/`writeNote`/`readLinkGraph`/`readSearchableNotes` + the identity→backing-file write resolver `resolveWriteTarget`), the pure **markdown** module (`markdown.ts`: `parseMarkdown`/`parseInline` → block model incl. headings, lists, and checkbox task lists, plus the inline grammar `**bold**`/`*italic*`/`` `code` ``/`[[wikilinks]]` with the wikilink parsing primitives `parseWikiLink`/`extractWikiLinks` keeping the literal `[[…]]` so notes round-trip, plus the deterministic+idempotent canonical serializer `serializeMarkdown` and the `canonicalizeMarkdown` = `serialize∘parse` shorthand, plus the optional YAML **frontmatter** subset (`parseFrontmatter` → `Frontmatter{title?,tags,created?,updated?,extra?}`, re-emitted first by `serializeMarkdown` in canonical key order — `title`/`created`/`updated`/sorted `extra`/`tags` — and only when non-empty, so frontmatter-free notes round-trip byte-identically); no DOM), the **sync** module (`sync.ts`: the pure autosave+squash state machine `NoteSync` driven through `onEdit`/`tick`/`flush`/`onFocusLeave`/`onIdle`/`onQuit` against an injected `SyncClock` + the narrow `WipSyncEngine`/`WipGitEngine` seam, `wipBranchName`, the `SYNC_PATH`/`NoteSyncRequest`/`NoteSyncResponse`/`SyncAction` HTTP contract, and the server-side dispatcher `applyNoteSync`; no push, by construction), the **note-mutation** module (`note-mutation.ts`: the pure create/rename/move planner `planCreateNote`/`planRenameNote`/`planMoveNote` → a `NoteMutation` of backing-file `moves`/`creates`/`removes`, carrying the leaf↔parent transition — promotion on first child, automatic+symmetric collapse on last child, root never collapses — and whole-subtree moves as one `dir` rename; the narrow `MutatingGitEngine.applyNoteMutation` seam + the `createNote`/`renameNote`/`moveNote` compositions; `slugifyNoteName`; `applyNoteMutationToFiles` for pure testing; the `NOTE_CREATE_PATH`/`NOTE_RENAME_PATH`/`NOTE_MOVE_PATH` HTTP contracts + `NoteMutationError` for 400-class rejections), and the **wikilink** module (`wikilink.ts`: the pure title resolver + link graph — `normalizeTitle`/`buildTitleIndex`/`resolveTitle`/`resolveWikiLink` resolve a `[[Note Name]]` to a note `path` by title case-/whitespace-insensitively, `null` = broken; `buildLinkGraph` extracts+resolves every note's wikilinks into deduped, sorted note→note `edges` + `broken` links, self-links ignored; the `LINKS_PATH`/`LinkGraphResponse` HTTP contract), and the **attachment** module (`attachment.ts`: the embedded-media write contract — embedded images/files are stored as real repo files under `ASSETS_DIR` (`assets/`) and referenced from Markdown by repo-relative path (`![alt](assets/diagram.png)`); the pure `slugifyAttachmentName` (kebab-case, extension-preserving, `file` fallback) + the `writeAttachment` composition over the narrow `AttachmentGitEngine.writeAttachmentFile` seam (the Node side owns byte-decoding, path-escape guarding, the commit, and collision-avoiding `-1`/`-2` suffixing — an IO concern), plus the `ATTACHMENT_PATH`/`AttachmentUploadRequest` (base64 `dataBase64`, plain-JSON, no multipart)/`AttachmentResponse` HTTP contract; no Node/DOM). The **search-index** module (`search-index.ts`: the pure search pipeline — `chunkNote` (reuses `parseMarkdown` to flatten a note's canonical Markdown into bounded, title-aware `NoteChunk`s), `cosineSimilarity`/`rankChunks`/`chunksToResults`, the keyword/filename term scorer `keywordSearch`, the narrow `Embedder` (`embed(text)→vector`) + `VectorStore` (`upsertNote`/`deleteNote`/`queryNearest`/`clear`) seams, the index ops `indexNote`/`removeNoteFromIndex`/`rebuildIndex`, `semanticSearch` (embed query → `queryNearest`, dedupe best chunk per note), and the fallback policy `runSearch` (semantic, else keyword — when asked, unavailable, thrown, or empty — reporting the `mode` that ran); plus the `InMemoryVectorStore` + deterministic `createHashingEmbedder` reference impls used as offline test doubles, and the `SEARCH_PATH`/`SearchRequest`/`SearchResponse`/`SearchResult` HTTP contract; no Node/DOM).
- `packages/ui` — Vite React SPA (`@stout/ui`). Builds to `dist/`, which the server serves statically. Talks to the server only via `/api/*`. The center panel renders a note through the swappable **Editor seam** (`editor.ts`: the `EditorComponent` "Markdown in, change events out" contract + pure Markdown↔ProseMirror-JSON bridge (incl. an image-only paragraph `![alt](assets/…)` ↔ a ProseMirror `image` node, translating the stored repo-relative path ↔ the hosted `/assets/…` src) + the pure wikilink helpers `wikiLinkQuery`/`filterTitles`/`scanWikiLinks` and the `WikiLinkContext` it renders against), with `TipTapEditor.tsx` as the default TipTap implementation (live formatting + checkboxes + embedded images via `@tiptap/extension-image`). It also renders **wikilinks**: a ProseMirror decoration (`wikilink-decoration.ts`) paints `[[links]]` in place (dashed+red when broken) without touching the literal Markdown, clicking a resolved link navigates, and typing `[[` opens a title-autocomplete popup (`WikiLinkSuggestions`); links resolve client-side off the loaded tree (`App` builds a `buildTitleIndex`/`resolveTitle` `WikiLinkContext`, no `/api/links` fetch). Editor `onChange` drives the autosave+squash session via the `useNoteSync` hook (`App.tsx`), which runs the `@stout/core` `NoteSync` machine over the HTTP `WipSyncEngine` adapter in `sync-client.ts` (`createHttpWipEngine`/`postNoteSync` → `POST /api/note/sync`): debounced edits autosave to the note's `wip/<note>` branch and the session squash-merges into `main` on focus-leave/hide/unload/note-switch. (The commit-on-save `postNote` → `POST /api/note` path still exists for explicit saves.) Above the editor, the center panel renders the note's **frontmatter**: a title heading (the frontmatter `title`, else the file-derived note title) plus `#tag` chips, and an **attachment upload** control (`AttachmentUpload`) that base64-encodes a picked image and posts it through `attachment-client.ts` (`postAttachment` → `POST /api/attachment`), then appends the returned `![alt](assets/…)` reference to the note body. The editor itself only ever sees the frontmatter-stripped body: `LoadedNote` (keyed on `note.path`) splits the frontmatter off, holds the body as local state, and recomposes (`recomposeNote`) frontmatter + body before feeding `useNoteSync`. The left nav also exposes **note mutations** — create (`+`), rename, and move affordances per node — which post through `mutation-client.ts` (`postNoteCreate`/`postNoteRename`/`postNoteMove` → `POST /api/note/create|rename|move`), then reselect the returned note and bump a reload token so `useTree` (lifted to `App`) refetches the reshaped tree. The center column also hosts **search** (`SearchPanel`/`useSearch` in `App.tsx`): a debounced search box that, only on non-empty input, queries `GET /api/search` through `search-client.ts` (`getSearch`), lists the ranked `SearchResult`s (title + snippet, labelled with the `mode` that ran) and, on click, `setSelected`s that note — reusing the same navigation as the tree and wikilinks.
- `apps/server` — Express host (`@stout/server`). Bootstraps Postgres, runs migrations, exposes `/api/health`, and serves the UI build. Entry: `src/index.ts`.
- `apps/electron` — minimal Electron shell (`@stout/electron`) loading the server-hosted UI.

### Server seams (keep these testable)

- `apps/server/src/migrate.ts` — migration runner decoupled via the `MigrationStore` interface (in-memory store in tests, pg-backed in prod). Add migrations to `src/migrations.ts`.
- `apps/server/src/db.ts` — `bootstrapDatabase` creates/uses the dedicated `stout` database and enables the `vector` extension; `PgMigrationStore` is the pg-backed ledger.
- `apps/server/src/git-engine.ts` — Node side of `core/git-engine`. `ensureWorkspaceRepo` initializes the bare repo + working clone (seeded with a starter `_index.md`) on first boot; `NodeGitEngine` implements the `@stout/core` `WipGitEngine` seam by shelling out to git. Commit-on-save / reads: `git ls-files` (`listNoteFiles`), reading a single note file off the working clone path-escape-guarded (`readNoteFile`), and writing + committing an edited note to `main` (`writeNoteFile`: `git add` + `git commit`, path-escape-guarded, skips no-op saves so there are no empty commits). Wip lifecycle (autosave+squash): `commitToWip` (checkout/create `wip/<note>` from `main`, write the backing file, commit — skipping no-ops), `squashMergeWipToMain` (`git merge --squash` + commit → one plain, linear `main` commit per session), and `deleteWip`; every wip op snaps the clone back to a clean `main` (`checkout -f main`) and nothing pushes, so wip branches stay local. Note mutations (create/rename/move): `applyNoteMutation` runs a `@stout/core` plan's `moves` (`git mv`, dirs included), `creates`, and `removes` as one atomic `main` commit, path-escape-guarded, rolling back (`git reset --hard` + `git clean -fd`) on any error. Attachments (`writeAttachmentFile`): write the given bytes near `assets/<slug>` in the working clone, resolve a collision-free name (`uniqueAttachmentPath`, `-1`/`-2` suffix), and `git add` + commit on `main`, path-escape-guarded, returning the final stored path (base64 decoding happens at the `index.ts` wiring). The pure file→tree / identity→file mapping, the canonicalize-then-persist composition (`writeNote`), the wip-branch slug (`wipBranchName`), the link-graph composition (`readLinkGraph`, which reuses the read seam — no new engine method), the searchable-corpus read (`readSearchableNotes`, likewise over the read seam), and the mutation planner stay in `core`. Requires the `git` binary at runtime (added to the Docker runner image).
- `apps/server/src/app.ts` — `createApp({ getHealth, getTree?, getNote?, saveNote?, syncNote?, createNote?, renameNote?, moveNote?, getLinks?, saveAttachment?, search?, assetsDir?, uiDir? })` takes injectable deps so HTTP behavior is tested without a live DB or repo. `getTree` is wired in prod to `readNoteTree(NodeGitEngine)` and exposed at `GET /api/tree` (contract `TREE_PATH`/`NoteTreeResponse`); `getNote` is wired to `readNote(NodeGitEngine, path)` and exposed at `GET /api/note?path=<identity>` (contract `NOTE_PATH`/`NoteContentResponse`, 404 when the note is missing); `saveNote` is wired to `writeNote(NodeGitEngine, path, markdown)` and exposed at `POST /api/note` (contract `NOTE_PATH`/`NoteSaveRequest` body → canonical `NoteContentResponse`, 400 when `markdown` is absent); `syncNote` is wired to `applyNoteSync(NodeGitEngine, request)` and exposed at `POST /api/note/sync` (contract `SYNC_PATH`/`NoteSyncRequest` → `NoteSyncResponse`, 400 on an unknown action or an `autosave` missing `markdown`); `createNote`/`renameNote`/`moveNote` are wired to the `@stout/core` compositions over `NodeGitEngine` and exposed at `POST /api/note/create|rename|move` (contracts `NOTE_CREATE_PATH`/`NOTE_RENAME_PATH`/`NOTE_MOVE_PATH` → `NoteMutationResponse`, 400 on a missing field or a `NoteMutationError`); `getLinks` is wired to `readLinkGraph(NodeGitEngine)` and exposed at `GET /api/links` (contract `LINKS_PATH`/`LinkGraphResponse`); `saveAttachment` is wired to `writeAttachment(NodeGitEngine, name, bytes)` (the `index.ts` wiring base64-decodes the request) and exposed at `POST /api/attachment` (`express.json({ limit: "25mb" })`, contract `ATTACHMENT_PATH`/`AttachmentUploadRequest` → `AttachmentResponse`, 400 on a missing `name`/`dataBase64`, 500 on an engine error), and stored attachments are served read-only from `assetsDir` at `GET /assets/*` (`express.static`, mounted before the SPA fallback); `search` is wired to `createSearchService(...).search` and exposed at `GET /api/search?q=&limit=&mode=` (contract `SEARCH_PATH`/`SearchRequest` → `SearchResponse`, parsing `q`→query, an in-range `limit`, and a `keyword`/`semantic` `mode`; only mounted when a searcher is injected; 500 on an engine error). All contracts live in `@stout/core`.

- `apps/server/src/embedder.ts` — the local embedding model loader. `loadEmbedder()` lazily imports `@xenova/transformers` (an **optional** dep, via a computed specifier so it is neither typechecked nor bundled) and builds an `Embedder` over `Xenova/all-MiniLM-L6-v2` (384-dim, in-process ONNX, no network/API); on any failure it falls back to the `@stout/core` `createHashingEmbedder` at the same `EMBEDDING_DIMENSIONS` (384), so a missing/unbuildable model degrades search rather than crashing.
- `apps/server/src/search-store.ts` — `PgVectorStore`, the Node/pg side of the `core/search-index` `VectorStore` seam over the `note_chunks` table (migration 2). `upsertNote` is an atomic delete-then-insert per note; `queryNearest` ranks in the DB via pgvector cosine distance (`<=>`, ivfflat index), returning `score = 1 - distance`; `deleteNote`/`clear` prune.
- `apps/server/src/search.ts` — `createSearchService({ engine, store, embedder })`, the composition that wires the `core/search-index` pipeline to a concrete engine/store/embedder. Exposes `search` (`runSearch` with the keyword fallback; `loadNotes` reads the corpus lazily via `readSearchableNotes`), `reindexNote` (re-embed one note by identity — or drop it from the index if gone — reading the corpus so its title matches a full rebuild), `removeNote`, and `rebuild` (re-embed the whole repo). Index updates are best-effort: `index.ts` fires them on commit (reindex after save/squash/create, rebuild after rename/move and on boot) as `void`-and-log, never failing the user's edit.
- `apps/server/src/migrations.ts` — migration **2** (`note-search-index`) creates the `note_chunks` table (`note_path`, `chunk_index`, `title`, `content`, `embedding vector(384)`) + a `note_path` btree index + an ivfflat cosine index on `embedding` (the `vector` extension is enabled in `bootstrapDatabase`).

## Dev workflow

- Install: `pnpm install`
- Build / test / typecheck all packages: `pnpm build`, `pnpm test`, `pnpm typecheck` (Turborepo, respects the `core → ui → server` build order).
- Run the server (needs built packages + a reachable Postgres): `pnpm start`.
- Full local stack: `docker compose up` (brings up a `pgvector` Postgres + the app container).

### Configuration

- `DATABASE_URL` points at the Postgres **maintenance** database (e.g. `.../postgres`); the server creates/switches to the dedicated `STOUT_DB_NAME` (default `stout`) itself. `STOUT_DATA_DIR` (default `data`, `/data` in the container) holds the note repo: a bare repo (`repo.git`) + working clone (`clone`). `PORT` defaults to `3000`. See `.env.example`.

### Environment notes

- pnpm 11 reads build-script allowlists and run settings from `pnpm-workspace.yaml` (`onlyBuiltDependencies`, `verifyDepsBeforeRun`, `strictDepBuilds`), not `.npmrc`.
- Tests are offline by design: the deep/seam logic (migration runner, health round-trip) is tested against in-memory/injected fakes, not a live Postgres.
