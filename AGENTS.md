# AGENTS.md

## Project layout

pnpm + Turborepo monorepo (the "walking skeleton", issue #1):

- `packages/core` — runtime-agnostic domain contracts (`@stout/core`). Pure, no Node/DOM deps. Holds shared types like `HealthStatus`, the **note-tree** mapper (`note-tree.ts`: file set → unified tree, pure), the **note-content** read/write contract (`note-content.ts`: `NOTE_PATH` + `NoteContentResponse`/`NoteSaveRequest` + the pure identity→backing-file resolver `noteFileCandidates`), the **git-engine** read/write seam (`git-engine.ts`: `GitEngine`/`WritableGitEngine` interfaces + `readNoteTree`/`readNote`/`writeNote`/`readLinkGraph` + the identity→backing-file write resolver `resolveWriteTarget`), the pure **markdown** module (`markdown.ts`: `parseMarkdown`/`parseInline` → block model incl. headings, lists, and checkbox task lists, plus the inline grammar `**bold**`/`*italic*`/`` `code` ``/`[[wikilinks]]` with the wikilink parsing primitives `parseWikiLink`/`extractWikiLinks` keeping the literal `[[…]]` so notes round-trip, plus the deterministic+idempotent canonical serializer `serializeMarkdown` and the `canonicalizeMarkdown` = `serialize∘parse` shorthand; no DOM), the **sync** module (`sync.ts`: the pure autosave+squash state machine `NoteSync` driven through `onEdit`/`tick`/`flush`/`onFocusLeave`/`onIdle`/`onQuit` against an injected `SyncClock` + the narrow `WipSyncEngine`/`WipGitEngine` seam, `wipBranchName`, the `SYNC_PATH`/`NoteSyncRequest`/`NoteSyncResponse`/`SyncAction` HTTP contract, and the server-side dispatcher `applyNoteSync`; no push, by construction), the **note-mutation** module (`note-mutation.ts`: the pure create/rename/move planner `planCreateNote`/`planRenameNote`/`planMoveNote` → a `NoteMutation` of backing-file `moves`/`creates`/`removes`, carrying the leaf↔parent transition — promotion on first child, automatic+symmetric collapse on last child, root never collapses — and whole-subtree moves as one `dir` rename; the narrow `MutatingGitEngine.applyNoteMutation` seam + the `createNote`/`renameNote`/`moveNote` compositions; `slugifyNoteName`; `applyNoteMutationToFiles` for pure testing; the `NOTE_CREATE_PATH`/`NOTE_RENAME_PATH`/`NOTE_MOVE_PATH` HTTP contracts + `NoteMutationError` for 400-class rejections), and the **wikilink** module (`wikilink.ts`: the pure title resolver + link graph — `normalizeTitle`/`buildTitleIndex`/`resolveTitle`/`resolveWikiLink` resolve a `[[Note Name]]` to a note `path` by title case-/whitespace-insensitively, `null` = broken; `buildLinkGraph` extracts+resolves every note's wikilinks into deduped, sorted note→note `edges` + `broken` links, self-links ignored; the `LINKS_PATH`/`LinkGraphResponse` HTTP contract).
- `packages/ui` — Vite React SPA (`@stout/ui`). Builds to `dist/`, which the server serves statically. Talks to the server only via `/api/*`. The center panel renders a note through the swappable **Editor seam** (`editor.ts`: the `EditorComponent` "Markdown in, change events out" contract + pure Markdown↔ProseMirror-JSON bridge + the pure wikilink helpers `wikiLinkQuery`/`filterTitles`/`scanWikiLinks` and the `WikiLinkContext` it renders against), with `TipTapEditor.tsx` as the default TipTap implementation (live formatting + checkboxes). It also renders **wikilinks**: a ProseMirror decoration (`wikilink-decoration.ts`) paints `[[links]]` in place (dashed+red when broken) without touching the literal Markdown, clicking a resolved link navigates, and typing `[[` opens a title-autocomplete popup (`WikiLinkSuggestions`); links resolve client-side off the loaded tree (`App` builds a `buildTitleIndex`/`resolveTitle` `WikiLinkContext`, no `/api/links` fetch). Editor `onChange` drives the autosave+squash session via the `useNoteSync` hook (`App.tsx`), which runs the `@stout/core` `NoteSync` machine over the HTTP `WipSyncEngine` adapter in `sync-client.ts` (`createHttpWipEngine`/`postNoteSync` → `POST /api/note/sync`): debounced edits autosave to the note's `wip/<note>` branch and the session squash-merges into `main` on focus-leave/hide/unload/note-switch. (The commit-on-save `postNote` → `POST /api/note` path still exists for explicit saves.) The left nav also exposes **note mutations** — create (`+`), rename, and move affordances per node — which post through `mutation-client.ts` (`postNoteCreate`/`postNoteRename`/`postNoteMove` → `POST /api/note/create|rename|move`), then reselect the returned note and bump a reload token so `useTree` (lifted to `App`) refetches the reshaped tree.
- `apps/server` — Express host (`@stout/server`). Bootstraps Postgres, runs migrations, exposes `/api/health`, and serves the UI build. Entry: `src/index.ts`.
- `apps/electron` — minimal Electron shell (`@stout/electron`) loading the server-hosted UI.

### Server seams (keep these testable)

- `apps/server/src/migrate.ts` — migration runner decoupled via the `MigrationStore` interface (in-memory store in tests, pg-backed in prod). Add migrations to `src/migrations.ts`.
- `apps/server/src/db.ts` — `bootstrapDatabase` creates/uses the dedicated `stout` database and enables the `vector` extension; `PgMigrationStore` is the pg-backed ledger.
- `apps/server/src/git-engine.ts` — Node side of `core/git-engine`. `ensureWorkspaceRepo` initializes the bare repo + working clone (seeded with a starter `_index.md`) on first boot; `NodeGitEngine` implements the `@stout/core` `WipGitEngine` seam by shelling out to git. Commit-on-save / reads: `git ls-files` (`listNoteFiles`), reading a single note file off the working clone path-escape-guarded (`readNoteFile`), and writing + committing an edited note to `main` (`writeNoteFile`: `git add` + `git commit`, path-escape-guarded, skips no-op saves so there are no empty commits). Wip lifecycle (autosave+squash): `commitToWip` (checkout/create `wip/<note>` from `main`, write the backing file, commit — skipping no-ops), `squashMergeWipToMain` (`git merge --squash` + commit → one plain, linear `main` commit per session), and `deleteWip`; every wip op snaps the clone back to a clean `main` (`checkout -f main`) and nothing pushes, so wip branches stay local. Note mutations (create/rename/move): `applyNoteMutation` runs a `@stout/core` plan's `moves` (`git mv`, dirs included), `creates`, and `removes` as one atomic `main` commit, path-escape-guarded, rolling back (`git reset --hard` + `git clean -fd`) on any error. The pure file→tree / identity→file mapping, the canonicalize-then-persist composition (`writeNote`), the wip-branch slug (`wipBranchName`), the link-graph composition (`readLinkGraph`, which reuses the read seam — no new engine method), and the mutation planner stay in `core`. Requires the `git` binary at runtime (added to the Docker runner image).
- `apps/server/src/app.ts` — `createApp({ getHealth, getTree?, getNote?, saveNote?, syncNote?, createNote?, renameNote?, moveNote?, getLinks?, uiDir? })` takes injectable deps so HTTP behavior is tested without a live DB or repo. `getTree` is wired in prod to `readNoteTree(NodeGitEngine)` and exposed at `GET /api/tree` (contract `TREE_PATH`/`NoteTreeResponse`); `getNote` is wired to `readNote(NodeGitEngine, path)` and exposed at `GET /api/note?path=<identity>` (contract `NOTE_PATH`/`NoteContentResponse`, 404 when the note is missing); `saveNote` is wired to `writeNote(NodeGitEngine, path, markdown)` and exposed at `POST /api/note` (contract `NOTE_PATH`/`NoteSaveRequest` body → canonical `NoteContentResponse`, 400 when `markdown` is absent); `syncNote` is wired to `applyNoteSync(NodeGitEngine, request)` and exposed at `POST /api/note/sync` (contract `SYNC_PATH`/`NoteSyncRequest` → `NoteSyncResponse`, 400 on an unknown action or an `autosave` missing `markdown`); `createNote`/`renameNote`/`moveNote` are wired to the `@stout/core` compositions over `NodeGitEngine` and exposed at `POST /api/note/create|rename|move` (contracts `NOTE_CREATE_PATH`/`NOTE_RENAME_PATH`/`NOTE_MOVE_PATH` → `NoteMutationResponse`, 400 on a missing field or a `NoteMutationError`); `getLinks` is wired to `readLinkGraph(NodeGitEngine)` and exposed at `GET /api/links` (contract `LINKS_PATH`/`LinkGraphResponse`). All contracts live in `@stout/core`.

## Dev workflow

- Install: `pnpm install`
- Build / test / typecheck all packages: `pnpm build`, `pnpm test`, `pnpm typecheck` (Turborepo, respects the `core → ui → server` build order).
- Run the server (needs built packages + a reachable Postgres): `pnpm start`.
- Full local stack: `docker compose up` (brings up a `pgvector` Postgres + the app container).

### Configuration

- `DATABASE_URL` points at the Postgres **maintenance** database (e.g. `.../postgres`); the server creates/switches to the dedicated `STOUT_DB_NAME` (default `stout`) itself. `STOUT_DATA_DIR` (default `data`, `/data` in the container) holds the note repo: a bare repo (`repo.git`) + working clone (`clone`). `PORT` defaults to `3000`. See `.env.example`.

### Environment notes

- pnpm 11 reads build-script allowlists and run settings from `pnpm-workspace.yaml` (`onlyBuiltDependencies`, `verifyDepsBeforeRun`, `strictDepBuilds`), not `.npmrc`.
- Tests are offline by design: the deep/seam logic (migration runner, health round-trip) is tested against in-memory/injected fakes, not a live Postgres.
