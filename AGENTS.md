# AGENTS.md

## Project layout

pnpm + Turborepo monorepo (the "walking skeleton", issue #1):

- `packages/core` — runtime-agnostic domain contracts (`@stout/core`). Pure, no Node/DOM deps. Holds shared types like `HealthStatus`, the **note-tree** mapper (`note-tree.ts`: file set → unified tree, pure; a note's frontmatter `title` overrides its file-name-derived title via the optional `NoteFile.title`), the **note-content** read/write contract (`note-content.ts`: `NOTE_PATH` + `NoteContentResponse`/`NoteSaveRequest` + the pure identity→backing-file resolver `noteFileCandidates` and its inverse `noteIdentityForFile` (backing file → note identity)), the **git-engine** read/write seam (`git-engine.ts`: `GitEngine`/`WritableGitEngine` interfaces + `readNoteTree`/`readNote`/`writeNote`/`readLinkGraph`/`readSearchableNotes` + the identity→backing-file write resolver `resolveWriteTarget`), the pure **markdown** module (`markdown.ts`: `parseMarkdown`/`parseInline` → block model incl. headings, lists, and checkbox task lists, plus the inline grammar `**bold**`/`*italic*`/`` `code` ``/`[[wikilinks]]` with the wikilink parsing primitives `parseWikiLink`/`extractWikiLinks` keeping the literal `[[…]]` so notes round-trip, plus the deterministic+idempotent canonical serializer `serializeMarkdown` and the `canonicalizeMarkdown` = `serialize∘parse` shorthand, plus the optional YAML **frontmatter** subset (`parseFrontmatter` → `Frontmatter{title?,tags,created?,updated?,extra?}`, re-emitted first by `serializeMarkdown` in canonical key order — `title`/`created`/`updated`/sorted `extra`/`tags` — and only when non-empty, so frontmatter-free notes round-trip byte-identically); no DOM), the **sync** module (`sync.ts`: the pure autosave+squash state machine `NoteSync` driven through `onEdit`/`tick`/`flush`/`onFocusLeave`/`onIdle`/`onQuit` against an injected `SyncClock` + the narrow `WipSyncEngine`/`WipGitEngine` seam, `wipBranchName`, the `SYNC_PATH`/`NoteSyncRequest`/`NoteSyncResponse`/`SyncAction` HTTP contract, and the server-side dispatcher `applyNoteSync`; no push, by construction), the **note-mutation** module (`note-mutation.ts`: the pure create/rename/move planner `planCreateNote`/`planRenameNote`/`planMoveNote` → a `NoteMutation` of backing-file `moves`/`creates`/`removes`, carrying the leaf↔parent transition — promotion on first child, automatic+symmetric collapse on last child, root never collapses — and whole-subtree moves as one `dir` rename; the narrow `MutatingGitEngine.applyNoteMutation` seam + the `createNote`/`renameNote`/`moveNote` compositions; `slugifyNoteName`; `applyNoteMutationToFiles` for pure testing; the `NOTE_CREATE_PATH`/`NOTE_RENAME_PATH`/`NOTE_MOVE_PATH` HTTP contracts + `NoteMutationError` for 400-class rejections), and the **wikilink** module (`wikilink.ts`: the pure title resolver + link graph — `normalizeTitle`/`buildTitleIndex`/`resolveTitle`/`resolveWikiLink` resolve a `[[Note Name]]` to a note `path` by title case-/whitespace-insensitively, `null` = broken; `buildLinkGraph` extracts+resolves every note's wikilinks into deduped, sorted note→note `edges` + `broken` links, self-links ignored; the `LINKS_PATH`/`LinkGraphResponse` HTTP contract), and the **attachment** module (`attachment.ts`: the embedded-media write contract — embedded images/files are stored as real repo files under `ASSETS_DIR` (`assets/`) and referenced from Markdown by repo-relative path (`![alt](assets/diagram.png)`); the pure `slugifyAttachmentName` (kebab-case, extension-preserving, `file` fallback) + the `writeAttachment` composition over the narrow `AttachmentGitEngine.writeAttachmentFile` seam (the Node side owns byte-decoding, path-escape guarding, the commit, and collision-avoiding `-1`/`-2` suffixing — an IO concern), plus the `ATTACHMENT_PATH`/`AttachmentUploadRequest` (base64 `dataBase64`, plain-JSON, no multipart)/`AttachmentResponse` HTTP contract; no Node/DOM). The **search-index** module (`search-index.ts`: the pure search pipeline — `chunkNote` (reuses `parseMarkdown` to flatten a note's canonical Markdown into bounded, title-aware `NoteChunk`s), `cosineSimilarity`/`rankChunks`/`chunksToResults`, the keyword/filename term scorer `keywordSearch`, the narrow `Embedder` (`embed(text)→vector`) + `VectorStore` (`upsertNote`/`deleteNote`/`queryNearest`/`clear`) seams, the index ops `indexNote`/`removeNoteFromIndex`/`rebuildIndex`, `semanticSearch` (embed query → `queryNearest`, dedupe best chunk per note), and the fallback policy `runSearch` (semantic, else keyword — when asked, unavailable, thrown, or empty — reporting the `mode` that ran); plus the `InMemoryVectorStore` + deterministic `createHashingEmbedder` reference impls used as offline test doubles, and the `SEARCH_PATH`/`SearchRequest`/`SearchResponse`/`SearchResult` HTTP contract; no Node/DOM). The **token-store** module (`token-store.ts`: the secret-at-rest seam for the **hub token** — the narrow `TokenStore` (get/set/clear) the hub-sync orchestrator reads from, the `SecureStorage` OS-encryption seam shaped to match Electron's `safeStorage` (`isEncryptionAvailable`/`encryptString`→`Uint8Array`/`decryptString`), the `SecureFilePorts` blob-IO seam (`readBlob`/`writeBlob`/`removeBlob`), and the pure `createSecureFileTokenStore` composition that encrypts **before** disk and **refuses to persist** when encryption is unavailable — no plaintext fallback; plus `InMemoryTokenStore` for tests; no Node/DOM/electron) and the **hub-sync** module (`hub-sync.ts`: the runtime-agnostic clone-then-sync policy for the **hub** — `syncWithHub` reads the token, authenticates the URL, and clones on first run / pulls-then-pushes thereafter over the narrow `HubRemoteEngine` seam (`hasLocalClone`/`cloneFromHub`/`pullFromHub`/`pushToHub`); the pure credential maths `authenticateRemoteUrl` (token → `x-access-token:<token>@host` userinfo) / `stripRemoteCredentials` / `redactRemoteUrl`, all no-ops on non-HTTP URLs so a local-path "hub" works in offline tests; `HubConfig`/`DEFAULT_HUB_BRANCH`/`HubSyncResult`; the token is materialised into a URL only for a single git op, never returned/logged/persisted; no Node/DOM). The **conflict** module (`conflict.ts`: the pure multi-device **conflict policy** — `resolveNoteConflict` reconciles a note's three-way (base/local/incoming) **canonical Markdown** (canonicalized first, so formatting-only diffs never conflict): equal sides / one-sided changes → a `clean` result; otherwise a line-level **diff3** (LCS sync-points) **auto-merges** non-overlapping edits or, on a true overlap, keeps the **incoming** version on the note and preserves the **local** version as a sibling **conflict copy** leaf — named from the original + a UTC `formatConflictMarker` (`YYYYMMDD-HHmmss`), frontmatter `title` set, identity de-duped against `existing`; `applyConflictResolution` writes the resolution through the same `WritableGitEngine` seam and emits a `ConflictNotification`; zero data loss, no Node/DOM) and the **sync-cadence** module (`sync-cadence.ts`: the pure `SyncScheduler` deciding **when** to sync — runs an injected `SyncRunner` on five triggers `launch`/`reconnect`/`focus`/`timer`/`manual`, **single-flight** + **coalescing** (mid-flight triggers collapse to one follow-up, stronger wins: manual>reconnect>launch>timer>focus) + **throttling** (only `focus` rate-limited by `minIntervalMs`; deliberate triggers always run); `tick(now)` self-gates the periodic `timer` by `periodMs`, records each run's outcome in `runs`; `DEFAULT_MIN_SYNC_INTERVAL_MS`/`DEFAULT_SYNC_PERIOD_MS`, imports `SyncClock` from `sync.ts`; no DOM/timer). The **remote-sync** module (`remote-sync.ts`: the runtime-agnostic **external-remote boundary merge** policy — when the server is configured to track an external Git remote (e.g. GitHub) other actors can also write to, this reconciles its divergent history into `main` with the same `core/conflict` policy. The pure `reconcileNotesWithIncoming(notes, options)` decides per changed note from its three versions `{base, local, incoming}`: `incoming` absent → keep local (external deletions **not** propagated), `local` absent → adopt incoming (clean), canonically equal → skip, else defer to `resolveNoteConflict` (auto-merge or keep-both conflict copy), de-duping copies against existing+seen identities. `syncRemoteBoundary(engine, tokenStore, config, options)` orchestrates over the narrow `RemoteBoundaryEngine extends WritableGitEngine` seam (`fetchBoundary`/`changedNoteFiles`/`readFileAt`/`recordBoundaryMerge`/`pushBoundary`): authenticate the URL (reusing `authenticateRemoteUrl`), fetch, gather each changed note's base/local/incoming, reconcile, `applyConflictResolution` each to `main`, record a tree-preserving merge so the push fast-forwards, then push — or just publish to seed an empty remote; `RemoteConfig`/`DEFAULT_REMOTE_BRANCH`/`BoundaryNote`/`BoundaryFetch`/`RemoteSyncResult`. Git never merges content — the conflict policy does; the token is materialised into a URL only for the fetch/push, never returned/logged/persisted; no Node/DOM).
- `packages/ui` — Vite React SPA (`@stout/ui`). Builds to `dist/`, which the server serves statically. The whole `App` reads and writes through a single injectable `fetch`-shaped **data-source seam** (`AppProps.fetchImpl`, default the global `fetch`, threaded through every hook + `*-client.ts`): the server-backed web app and Electron leave it unset and talk to the live `/api/*` HTTP surface, while the offline PWA runtime injects an in-browser adapter that answers the same contracts locally (see "the offline runtime" at the end of this bullet). The center panel renders a note through the swappable **Editor seam** (`editor.ts`: the `EditorComponent` "Markdown in, change events out" contract + pure Markdown↔ProseMirror-JSON bridge (incl. an image-only paragraph `![alt](assets/…)` ↔ a ProseMirror `image` node, translating the stored repo-relative path ↔ the hosted `/assets/…` src) + the pure wikilink helpers `wikiLinkQuery`/`filterTitles`/`scanWikiLinks` and the `WikiLinkContext` it renders against), with `TipTapEditor.tsx` as the default TipTap implementation (live formatting + checkboxes + embedded images via `@tiptap/extension-image`). It also renders **wikilinks**: a ProseMirror decoration (`wikilink-decoration.ts`) paints `[[links]]` in place (dashed+red when broken) without touching the literal Markdown, clicking a resolved link navigates, and typing `[[` opens a title-autocomplete popup (`WikiLinkSuggestions`); links resolve client-side off the loaded tree (`App` builds a `buildTitleIndex`/`resolveTitle` `WikiLinkContext`, no `/api/links` fetch). Editor `onChange` drives the autosave+squash session via the `useNoteSync` hook (`App.tsx`), which runs the `@stout/core` `NoteSync` machine over the HTTP `WipSyncEngine` adapter in `sync-client.ts` (`createHttpWipEngine`/`postNoteSync` → `POST /api/note/sync`): debounced edits autosave to the note's `wip/<note>` branch and the session squash-merges into `main` on focus-leave/hide/unload/note-switch. (The commit-on-save `postNote` → `POST /api/note` path still exists for explicit saves.) Above the editor, the center panel renders the note's **frontmatter**: a title heading (the frontmatter `title`, else the file-derived note title) plus `#tag` chips, and an **attachment upload** control (`AttachmentUpload`) that base64-encodes a picked image and posts it through `attachment-client.ts` (`postAttachment` → `POST /api/attachment`), then appends the returned `![alt](assets/…)` reference to the note body. The editor itself only ever sees the frontmatter-stripped body: `LoadedNote` (keyed on `note.path`) splits the frontmatter off, holds the body as local state, and recomposes (`recomposeNote`) frontmatter + body before feeding `useNoteSync`. The left nav also exposes **note mutations** — create (`+`), rename, and move affordances per node — which post through `mutation-client.ts` (`postNoteCreate`/`postNoteRename`/`postNoteMove` → `POST /api/note/create|rename|move`), then reselect the returned note and bump a reload token so `useTree` (lifted to `App`) refetches the reshaped tree. The left navigation panel also hosts **search** (`SearchPanel`/`useSearch` in `App.tsx`): a debounced search box that, only on non-empty input, queries `GET /api/search` through `search-client.ts` (`getSearch`), lists the ranked `SearchResult`s (title + snippet, labelled with the `mode` that ran) and, on click, `setSelected`s that note — reusing the same navigation as the tree and wikilinks. The whole UI is a **three-panel workspace** (`App.tsx`): the left `NavPanel` (`data-testid="nav-panel"`, brand + "New Note" + search + tree), the center editor `main` (`editor-panel`), and the right contextual-utilities `UtilityPanel` (`utility-panel`), with a `MobileTabs` (`role="tab"` × Navigation/Editor/Context) switcher for the single-column mobile layout; `selectNote` sets the selected note and focuses the editor pane (so opening a note on mobile brings it into view). `useNote` and `useLinks` are lifted to `App` so one fetch each feeds both the editor and the utilities panel; `useLinks` (`GET /api/links` → `LinkGraph`, contract `LINKS_PATH`) degrades **gracefully** to the empty graph on 404/error. The `UtilityPanel` is composed from already-loaded data — **Details** (the selected note's tree node: identity/file/kind), **Outline** (`noteOutline` flattens the note body's headings via `parseMarkdown`), **Links** (backlinks/outbound/broken sliced from the link graph, rows navigate), and **System** (`HealthSection` — the `status`/`database`/`migration` health, moved here) — empty until a note is open. Styling is the **Technical Umber** design system (`DESIGN.md`, ADR 0009): `styles.css` holds the palette/type/radius/spacing tokens + the three-panel grid + components + the `@media (max-width:1023px)` single-column collapse, and is imported **only** from `main.tsx` (never `App.tsx`) so the shipped app is themed while unit tests render unstyled and assert on structure; layout is pure responsive CSS with no per-runtime fork (browser and Electron render identically), and fonts are named-first (Geist / Hanken Grotesk / JetBrains Mono) with a system fallback so nothing is fetched at runtime. The SPA is also an **offline-capable PWA** (ADR 0011): `vite.config.ts` runs `VitePWA` (`generateSW`, `autoUpdate`, manifest with name/`#111415` theme/`icon.svg`, workbox `navigateFallback` to `index.html` with an `/^\/api\//`+`/^\/assets\//` denylist so app-shell navigations are cached but data/asset requests always hit the network), `main.tsx` calls `registerSW({ immediate: true })` from `virtual:pwa-register`, `src/vite-env.d.ts` references the `vite/client`+`vite-plugin-pwa/client` types, and `public/icon.svg` is the app monogram; the build emits `dist/sw.js`+`dist/manifest.webmanifest`. For a **fully in-browser runtime** (no server), the **browser git engine** (`browser-git-engine.ts`) implements the very same `@stout/core` git-engine seam as `NodeGitEngine` by driving `isomorphic-git` over `@isomorphic-git/lightning-fs` (an IndexedDB-backed filesystem): `ensureBrowserRepo` idempotently inits `main` and seeds a starter note, and the pure path helpers in `browser-fs.ts` (`isSafeRepoPath`/`repoFilePath`/`ancestorDirs`/`toNoteFiles`, `BROWSER_WORKDIR`/`BROWSER_FS_NAME`) keep all filesystem maths testable off any real IndexedDB. Multi-device **conflicts** surface non-blockingly: `conflict-toast.tsx` renders a fixed-corner, `aria-live=polite` stack of `ConflictNotification` toasts (the `useConflictNotifications` hook holds the queue) with an "Open copy" action that navigates to the preserved conflict-copy note and a dismiss control; `App.tsx` wires it additively through an optional `onConflicts?: (notify) => void` prop (the server-backed web app leaves it unset). The `sync-cadence-controller.ts` adapts the `@stout/core` `SyncScheduler` to browser DOM events (focus/online/visibility/timer) for the PWA runtime. **The offline runtime** assembles these browser pieces into a running, server-free app without forking the UI: `browser-api.ts` (`createBrowserApiFetch`) is a `typeof fetch` that answers the same `/api/*` contracts locally by running the same `@stout/core` compositions against the IndexedDB `BrowserGitEngine` — `readNoteTree`/`readNote`/`writeNote`/`readLinkGraph`, `keywordSearch` over `readSearchableNotes`, and `applyNoteSync` over `createCommitOnSaveWipEngine` (offline is a single writer, so an autosave commits straight to `main` and squash/delete-wip are no-ops); note mutations and attachments return a graceful `501` until the browser engine grows those seams. `offline-sync.ts` (`createOfflineSyncRunner`) is the client-side `SyncRunner` the cadence controller drives — it reconciles with a remote and forwards each `ConflictNotification` to the toast sink, degrading to a well-behaved no-op when there is no remote. `offline-runtime.tsx` (`startOfflineApp`) is the composition root: `ensureBrowserRepo()` → `new BrowserGitEngine()` → `createBrowserApiFetch` → `createSyncController(createOfflineSyncRunner({ notify }))` → render `<App fetchImpl onConflicts/>`. One built SPA serves both runtimes: `main.tsx` picks at load time via `resolveRuntimeMode(location.search)` (`runtime.ts`), and the manifest's `start_url` is `/?runtime=offline` (`vite.config.ts`) so the installed PWA boots offline while the web app at `/` stays on the server. (The adapter, sync runner, and runtime selection are unit-tested offline against in-memory fakes; the IndexedDB engine and `startOfflineApp` need a real browser and are a documented follow-up.)
- `apps/server` — Express host (`@stout/server`). Bootstraps Postgres, runs migrations, exposes `/api/health`, and serves the UI build. Entry: `src/index.ts`. Ships a **second** entry `src/desktop.ts` (`tsup` builds both; `exports` map `.`→index, `./desktop`→desktop) — the Express-free `@stout/server/desktop` surface the Electron shell boots over a local clone (`startLocalWorkspace`).
- `apps/electron` — **local-first desktop** (`@stout/electron`). Runs the same `@stout/ui` SPA + same `/api/*` surface as the web server via `@stout/server/desktop`'s `startLocalWorkspace` over a **local** working clone (under the app's user-data dir), with an **in-memory** search index instead of Postgres (so health reports `database:false` as a healthy state) — no UI fork, window loads a loopback URL. `src/main.ts` bootstraps the repo (if `STOUT_HUB_URL` is set, `syncWithHub` over `NodeHubRemoteEngine` reading the **hub token** from the OS keychain; else/on failure `ensureWorkspaceRepo` for a local-only workspace) then opens the window. `src/safe-storage.ts` is the Electron adapter for the `@stout/core` token-store seams: `electronSecureStorage` over `safeStorage`, file ports at `userData/hub-token.bin`, `createDesktopTokenStore`, and `loadHubConfig` (reads `STOUT_HUB_URL`/`STOUT_HUB_BRANCH`). Depends on `@stout/server` (workspace); build externalizes `electron`/`@stout/server`/`@stout/core`. Packaging (electron-builder) is a documented follow-up — see ADR 0010.

### Server seams (keep these testable)

- `apps/server/src/migrate.ts` — migration runner decoupled via the `MigrationStore` interface (in-memory store in tests, pg-backed in prod). Add migrations to `src/migrations.ts`.
- `apps/server/src/db.ts` — `bootstrapDatabase` creates/uses the dedicated `stout` database and enables the `vector` extension; `PgMigrationStore` is the pg-backed ledger.
- `apps/server/src/git-engine.ts` — Node side of `core/git-engine`. `ensureWorkspaceRepo` initializes the bare repo + working clone (seeded with a starter `_index.md`) on first boot; `NodeGitEngine` implements the `@stout/core` `WipGitEngine` seam by shelling out to git. Commit-on-save / reads: `git ls-files` (`listNoteFiles`), reading a single note file off the working clone path-escape-guarded (`readNoteFile`), and writing + committing an edited note to `main` (`writeNoteFile`: `git add` + `git commit`, path-escape-guarded, skips no-op saves so there are no empty commits). Wip lifecycle (autosave+squash): `commitToWip` (checkout/create `wip/<note>` from `main`, write the backing file, commit — skipping no-ops), `squashMergeWipToMain` (`git merge --squash` + commit → one plain, linear `main` commit per session), and `deleteWip`; every wip op snaps the clone back to a clean `main` (`checkout -f main`) and nothing pushes, so wip branches stay local. Note mutations (create/rename/move): `applyNoteMutation` runs a `@stout/core` plan's `moves` (`git mv`, dirs included), `creates`, and `removes` as one atomic `main` commit, path-escape-guarded, rolling back (`git reset --hard` + `git clean -fd`) on any error. Attachments (`writeAttachmentFile`): write the given bytes near `assets/<slug>` in the working clone, resolve a collision-free name (`uniqueAttachmentPath`, `-1`/`-2` suffix), and `git add` + commit on `main`, path-escape-guarded, returning the final stored path (base64 decoding happens at the `index.ts` wiring). The pure file→tree / identity→file mapping, the canonicalize-then-persist composition (`writeNote`), the wip-branch slug (`wipBranchName`), the link-graph composition (`readLinkGraph`, which reuses the read seam — no new engine method), the searchable-corpus read (`readSearchableNotes`, likewise over the read seam), and the mutation planner stay in `core`. Requires the `git` binary at runtime (added to the Docker runner image).
- `apps/server/src/hub-engine.ts` — Node side of `core/hub-sync`: `NodeHubRemoteEngine` implements the `@stout/core` `HubRemoteEngine` seam by shelling `git` against the **hub** for the desktop's local clone (`hasLocalClone` = does `<clone>/.git` exist; `cloneFromHub` then `remote set-url origin <stripped>` + local identity + best-effort `checkout <branch>`; `pullFromHub` = `pull --no-rebase --no-edit`; `pushToHub` = `push <url> HEAD:<branch>`). The hub-remote counterpart to `NodeGitEngine` — the only other place that touches `git`. The token only ever appears in a single `git` argv (the authenticated URL), never persisted to `.git/config`.
- `apps/server/src/remote-engine.ts` — Node side of `core/remote-sync`: `NodeRemoteBoundaryEngine extends NodeGitEngine` implements the `@stout/core` `RemoteBoundaryEngine` seam by shelling `git` against a configurable **external remote** (e.g. GitHub) — `fetchBoundary` (`checkout -f main` + `fetch <url> <branch>`, `null` when the branch is absent = empty remote, else `FETCH_HEAD` tip + `merge-base` with `HEAD`), `changedNoteFiles` (`diff --name-only -z`, `.md` only), `readFileAt` (`show <ref>:<file>`, `null` if absent), `recordBoundaryMerge` (`merge -s ours --allow-unrelated-histories` — keeps the already-reconciled tree, just adds the external tip as a 2nd parent so the push fast-forwards), and `pushBoundary` (`push <url> HEAD:<branch>`). The external-remote counterpart to `NodeGitEngine` (it extends it, so one engine serves both the HTTP layer and the sync loop); the token only ever appears in a single `git` argv, never persisted to `.git/config`. Also `loadRemoteConfig(env)` → `RemoteSetup{config, token}|null` from `STOUT_REMOTE_URL`/`STOUT_REMOTE_BRANCH`/`STOUT_REMOTE_TOKEN` (bare aliases too); `null` = internal bare repo only. The sync loop itself is wired in `index.ts` (`startRemoteSyncLoop`: drives `syncRemoteBoundary` through the `core/sync-cadence` `SyncScheduler` on launch + a periodic timer, logs conflict copies + redacted URL, rebuilds search after a sync, never crashes on failure).
- `apps/server/src/desktop.ts` — the **Express-free** `@stout/server/desktop` entry the Electron shell boots. `startLocalWorkspace({ cloneDir, uiDir?, port?, host? })` wires the very same `createApp(...)` surface + built `@stout/ui` SPA the web server serves, but over a `NodeGitEngine` on a local clone and an **in-memory** search (`createHashingEmbedder` + `InMemoryVectorStore`, rebuilt in the background on start, best-effort on edits); health is `ok` with `database:false`; binds loopback (`127.0.0.1`) on an ephemeral port and returns a `LocalWorkspace` (`url` + `close()`). Re-exports the Node-side plumbing the bootstrap needs (`NodeGitEngine`/`ensureWorkspaceRepo`/`loadRepoPaths`/`RepoPaths`/`NodeHubRemoteEngine`/`resolveUiDir`) so Electron imports everything from one entry. The public surface drags in no `@types/express`, so the Electron app's declarations stay Express-free.
- `apps/server/src/app.ts` — `createApp({ getHealth, getTree?, getNote?, saveNote?, syncNote?, createNote?, renameNote?, moveNote?, getLinks?, saveAttachment?, search?, assetsDir?, uiDir? })` takes injectable deps so HTTP behavior is tested without a live DB or repo. `getTree` is wired in prod to `readNoteTree(NodeGitEngine)` and exposed at `GET /api/tree` (contract `TREE_PATH`/`NoteTreeResponse`); `getNote` is wired to `readNote(NodeGitEngine, path)` and exposed at `GET /api/note?path=<identity>` (contract `NOTE_PATH`/`NoteContentResponse`, 404 when the note is missing); `saveNote` is wired to `writeNote(NodeGitEngine, path, markdown)` and exposed at `POST /api/note` (contract `NOTE_PATH`/`NoteSaveRequest` body → canonical `NoteContentResponse`, 400 when `markdown` is absent); `syncNote` is wired to `applyNoteSync(NodeGitEngine, request)` and exposed at `POST /api/note/sync` (contract `SYNC_PATH`/`NoteSyncRequest` → `NoteSyncResponse`, 400 on an unknown action or an `autosave` missing `markdown`); `createNote`/`renameNote`/`moveNote` are wired to the `@stout/core` compositions over `NodeGitEngine` and exposed at `POST /api/note/create|rename|move` (contracts `NOTE_CREATE_PATH`/`NOTE_RENAME_PATH`/`NOTE_MOVE_PATH` → `NoteMutationResponse`, 400 on a missing field or a `NoteMutationError`); `getLinks` is wired to `readLinkGraph(NodeGitEngine)` and exposed at `GET /api/links` (contract `LINKS_PATH`/`LinkGraphResponse`); `saveAttachment` is wired to `writeAttachment(NodeGitEngine, name, bytes)` (the `index.ts` wiring base64-decodes the request) and exposed at `POST /api/attachment` (`express.json({ limit: "25mb" })`, contract `ATTACHMENT_PATH`/`AttachmentUploadRequest` → `AttachmentResponse`, 400 on a missing `name`/`dataBase64`, 500 on an engine error), and stored attachments are served read-only from `assetsDir` at `GET /assets/*` (`express.static`, mounted before the SPA fallback); `search` is wired to `createSearchService(...).search` and exposed at `GET /api/search?q=&limit=&mode=` (contract `SEARCH_PATH`/`SearchRequest` → `SearchResponse`, parsing `q`→query, an in-range `limit`, and a `keyword`/`semantic` `mode`; only mounted when a searcher is injected; 500 on an engine error). All contracts live in `@stout/core`.

- `apps/server/src/embedder.ts` — the local embedding model loader. `loadEmbedder()` lazily imports `@xenova/transformers` (an **optional** dep, via a computed specifier so it is neither typechecked nor bundled) and builds an `Embedder` over `Xenova/all-MiniLM-L6-v2` (384-dim, in-process ONNX, no network/API); on any failure it falls back to the `@stout/core` `createHashingEmbedder` at the same `EMBEDDING_DIMENSIONS` (384), so a missing/unbuildable model degrades search rather than crashing.
- `apps/server/src/search-store.ts` — `PgVectorStore`, the Node/pg side of the `core/search-index` `VectorStore` seam over the `note_chunks` table (migration 2). `upsertNote` is an atomic delete-then-insert per note; `queryNearest` ranks in the DB via pgvector cosine distance (`<=>`, ivfflat index), returning `score = 1 - distance`; `deleteNote`/`clear` prune.
- `apps/server/src/search.ts` — `createSearchService({ engine, store, embedder })`, the composition that wires the `core/search-index` pipeline to a concrete engine/store/embedder. Exposes `search` (`runSearch` with the keyword fallback; `loadNotes` reads the corpus lazily via `readSearchableNotes`), `reindexNote` (re-embed one note by identity — or drop it from the index if gone — reading the corpus so its title matches a full rebuild), `removeNote`, and `rebuild` (re-embed the whole repo). Index updates are best-effort: `index.ts` fires them on commit (reindex after save/squash/create, rebuild after rename/move and on boot) as `void`-and-log, never failing the user's edit.
- `apps/server/src/migrations.ts` — migration **2** (`note-search-index`) creates the `note_chunks` table (`note_path`, `chunk_index`, `title`, `content`, `embedding vector(384)`) + a `note_path` btree index + an ivfflat cosine index on `embedding` (the `vector` extension is enabled in `bootstrapDatabase`).

## Dev workflow

- Install: `pnpm install`
- Build / test / typecheck all packages: `pnpm build`, `pnpm test`, `pnpm typecheck` (Turborepo, respects the `core → ui → server` build order).
- Run the server (needs built packages + a reachable Postgres): `pnpm start`.
- Full local stack: `docker compose up` (brings up a `pgvector` Postgres + the app container).

### Configuration

- `DATABASE_URL` points at the Postgres **maintenance** database (e.g. `.../postgres`); the server creates/switches to the dedicated `STOUT_DB_NAME` (default `stout`) itself. `STOUT_DATA_DIR` (default `data`, `/data` in the container) holds the note repo: a bare repo (`repo.git`) + working clone (`clone`). `PORT` defaults to `3000`. See `.env.example`.
- **Electron desktop (all optional):** `STOUT_HUB_URL` enables **hub sync** — the desktop clones it on first run and pulls-then-pushes thereafter; unset ⇒ fully-local, no network. `STOUT_HUB_BRANCH` is the branch to sync (default `main`). The **hub token** is **not** an env var: it lives encrypted in the OS keychain (`safeStorage`), never in plaintext. The desktop's own clone lives under the app's user-data dir, not `STOUT_DATA_DIR`.

### Environment notes

- pnpm 11 reads build-script allowlists and run settings from `pnpm-workspace.yaml` (`onlyBuiltDependencies`, `verifyDepsBeforeRun`, `strictDepBuilds`), not `.npmrc`.
- Tests are offline by design: the deep/seam logic (migration runner, health round-trip) is tested against in-memory/injected fakes, not a live Postgres.
